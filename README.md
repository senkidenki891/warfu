# warfu

`warfu` — прототип тактического RTwP-движка с разделением на симуляцию (`src/sim`) и пользовательский интерфейс (`src/ui`).

## Запуск

1. Установите зависимости:
   ```bash
   npm install
   ```
2. Запустите тесты:
   ```bash
   npm test
   ```
3. Для разработки тестов в watch-режиме:
   ```bash
   npm run test:watch
   ```

## Игровой цикл RTwP

Игровой цикл организован как непрерывная симуляция с дискретными тиками:

1. **Input phase**: UI получает команды игрока (движение, огонь, миссии).
2. **Command validation**: симуляция проверяет доступность приказа и формирует order queue.
3. **Delay/reliability resolution**: рассчитывается задержка исполнения и вероятность успешной передачи.
4. **Execution phase**: юниты выполняют действия (перемещение, обнаружение, огонь, снабжение).
5. **State publish**: новый state отдается в UI как read-only snapshot.

## Command delay

`command delay` моделирует время прохождения приказа по C2-цепочке и влияние качества связи.

- Базовая задержка (`baseDelay`) зависит от типа подразделения/доктрины.
- Надежность (`reliability`) определяет шанс принятия приказа.
- Отказ/помеха увеличивает фактическую задержку и может приводить к повтору приказа.

## Добавление юнита/техники/оружия/местности

Рекомендуемый workflow:

1. Описать сущность в данных симуляции (`src/sim`) через типы/конфигурации.
2. Добавить поведенческую логику (стоимость движения, поражение, обнаружение, боекомплект).
3. Добавить тесты в `src/tests` на:
   - корректность расчетов,
   - крайние случаи,
   - интеграцию с существующими механиками.
4. Подключить отображение в `src/ui` без дублирования правил симуляции.

## Сценарии

Сценарий включает:

- карту (проходимость, укрытие, ключевые зоны),
- стороны конфликта и начальные группы,
- условия победы/поражения,
- лимиты времени,
- доступные fire-support миссии.

Сценарные данные должны быть декларативными и не смешиваться с процедурной логикой UI.

## Damage/components

Модель урона учитывает:

- **общую прочность** платформы (`hp`),
- **подсистемы** (`components`: двигатель, оптика, вооружение, связь),
- деградацию боевой эффективности по мере повреждения компонентов.

Пример: попадание в двигатель снижает и `hp`, и ресурс компонента `engine`, что уменьшает подвижность.

## Contacts/concealment

Контакт хранит уровень уверенности (`confidence`) и деградирует во времени:

- со временем без подтверждения,
- быстрее при высоком `concealment` цели,
- до полной потери контакта (confidence → 0).

Это позволяет моделировать «призрачные» цели, устаревшую разведданную и необходимость повторной разведки.

## Air/artillery missions

### Artillery

- Запрос огня имеет задержку: `requestAt + prepDelay + commsDelay`.
- Точность и эффект могут зависеть от наблюдателя, связи и типа боеприпаса.

### Air missions

- Авиаудар проходит проверку ПВО (`interceptionChance`).
- При успешном перехвате часть урона гасится.
- При неуспешном — удар наносится в полном объеме.

## Architecture

### Граница `src/ui` и `src/sim`

**`src/sim`** (domain/simulation):

- детерминированные расчеты правил,
- модели состояния и переходы,
- pathfinding, LOS, damage, morale, logistics, fire-support.

**`src/ui`** (presentation/input):

- визуализация состояния,
- сбор пользовательского ввода,
- отображение контактов/подсказок/таймлайна,
- отправка команд в `src/sim` через узкий API.

### Правило зависимости

- `src/sim` **не должен** импортировать `src/ui`.
- `src/ui` может импортировать контракты/типы из `src/sim`.
- Вся игровая логика должна тестироваться через `src/tests` и не зависеть от рендеринга.
